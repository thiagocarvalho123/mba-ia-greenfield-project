### SI-03.8 — Video Worker Bootstrap and Processing Job Handler
- **Status:** done
- **Tests:** `video-processing.consumer.spec.ts` (2 unit tests: `handle` sets `status: 'ready'` with probed duration and captured thumbnail key on success, `handleFailure` sets `status: 'failed'` with the given reason) and `queue.module.spec.ts` (1 compilation test: `QueueModule` resolves both `QueueService` and the `PG_BOSS` token) plus an updated `queue.service.spec.ts` assertion for the new dead-letter queue creation — full suite 198/198 unit+integration and 70/70 e2e passing (both reran serially with `--runInBand` for a clean confirmation, since the default parallel Jest run showed transient cross-suite DB contention between `auth.e2e-spec.ts` and `videos.e2e-spec.ts` sharing one Postgres instance — pre-existing to this SI, not a regression), `npx tsc --noEmit` clean, lint clean on all newly-authored code.
- **Spec deviation:** the phase document describes `boss.onFail(QUEUE_NAMES.VIDEO_PROCESSING, ...)` for handling exhausted-retry failures, but this API does not exist in the installed `pg-boss` v12 (confirmed by reading its compiled `.d.ts`/`.js` sources — `PgBossEventMap` only exposes `error | warning | wip | stopped | bam | flow`). Implemented the equivalent behavior using pg-boss v12's real mechanism instead: `boss.createQueue(QUEUE_NAMES.VIDEO_PROCESSING, { deadLetter: QUEUE_NAMES.VIDEO_PROCESSING_DLQ })`, with a second `boss.work()` handler registered on the dead-letter queue (`{ includeMetadata: true }`) to read `job.output.message` and mark the video `failed` with the failure reason.
- **Bugs found via manual smoke-test (not caught by unit tests or `tsc`):** started the `video-worker` Compose service directly (`docker compose up -d video-worker`, after `npm run build`) to verify the "container registers a handler without requiring `nestjs-api` to be running" acceptance criterion, and it failed to boot twice:
  1. `UnknownDependenciesException` for `PG_BOSS` in `VideoProcessingConsumer` — `QueueModule` only exported `QueueService`, not the `PG_BOSS` token that the consumer needs directly to call `.work()`. Fixed by adding `PG_BOSS` to `QueueModule`'s `exports`, and added `queue.module.spec.ts` (a compilation test that resolves both tokens) so this class of export bug is caught automatically going forward.
  2. `TypeORMError: Entity metadata for Video#channel/Channel#user was not found` — `WorkerModule`'s `TypeOrmModule.forRootAsync` uses `autoLoadEntities: true`, which only picks up entities registered via some `forFeature()` call *within the bootstrapped module graph*; since `WorkerModule` only registered `Video`, TypeORM couldn't resolve `Video`'s `@ManyToOne(() => Channel)` relation (and then `Channel`'s `@OneToOne(() => User)` relation once `Channel` was added). Fixed by registering the full transitive closure via `TypeOrmModule.forFeature([Video, Channel, User])` (`User` has no further outward relations, so the closure stops there). This class of bug is inherent to a real DB connection attempt and would not be caught by a compilation test using a mocked `DataSource` — confirming the earlier decision not to write a `worker.module.spec.ts` was correct, but underscoring why the manual smoke-test step is required for any module doing a real `TypeOrmModule.forRootAsync`.
- After both fixes, `docker compose up -d video-worker` boots cleanly, connects to the real Postgres, and logs "Video processing worker registered and running" without `nestjs-api` needing to be running — closing out the deferred SI-03.1 observation.

## Compliance addendum (2026-07-05)

All 8 SIs above were marked "done" before a literal-compliance audit against `plan.txt` and `docs/project-plan.md` surfaced 4 findings, all now fixed:

1. **Short + unique URL (`project-plan.md` §4 "URL curta e única"):** TD-05 originally chose the UUID PK directly as the public id, on the incorrect claim that no requirement asked for a short URL. TD-04 amended with this scope note.
